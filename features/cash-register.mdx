# Cash Register Management

The Cash Register module manages daily cash drawer operations, tracking opening balances, sales, and closing reconciliation with automatic variance calculation.

## Overview

This component provides a simple interface for opening and closing cash registers, tracking expected vs. actual balances, and maintaining a history of register operations.

**Location**: `~/workspace/source/src/app/components/pages/cash-register/cash-register.component.ts`

<Note>
  This is a foundational implementation that tracks basic cash register operations. It currently uses local state and can be extended to integrate with backend services.
</Note>

## Closing a Cash Register

<Steps>
  <Step title="Initiate Closing">
    User clicks the "Cerrar Caja" button to start the closing process.
  </Step>
  
  <Step title="Count Cash and Cards">
    Physically count the drawer and enter amounts:
    - **Actual Cash**: Physical cash in the drawer
    - **Actual Card**: Total card transactions (from payment terminal)
    
    ```typescript
    actualCash: number | null = null;
    actualCard: number | null = null;
    ```
  </Step>
  
  <Step title="Calculate Variance">
    The system automatically calculates the difference:
    
    ```typescript
    handleCloseCashRegister(): void {
        if (!this.activeCashRegister || 
            this.actualCash === null || 
            this.actualCard === null) {
            return;
        }
        
        const totalActual = this.actualCash + this.actualCard;
        const difference = totalActual - 
            (this.activeCashRegister.expectedBalance || 0);
        
        this.cashRegisters = this.cashRegisters.map((cr) =>
            cr.id === this.activeCashRegister?.id
                ? {
                    ...cr,
                    status: "closed",
                    closedAt: new Date().toISOString(),
                    actualBalance: totalActual,
                    difference: difference
                  }
                : cr
        );
        
        console.log("Cerrando caja - Diferencia:", difference);
        this.showCloseDialog = false;
        this.actualCash = null;
        this.actualCard = null;
        this.updateActiveCashRegister();
    }
    ```
    **Source**: `cash-register.component.ts:107-132`
  </Step>
  
  <Step title="Review Results">
    The closed register now shows:
    - Opening balance
    - Expected balance
    - Actual balance
    - **Difference** (positive = overage, negative = shortage)
  </Step>
</Steps>

    const now = Date.now();

## Variance Analysis

### Understanding Differences

<Tabs>
  <Tab title="Positive Difference">
    **Overage**: Actual balance is higher than expected.
    
    Possible causes:
    - Cashier error (gave too little change)
    - Unrecorded sale
    - Counting error
  </Tab>
  
  <Tab title="Negative Difference">
    **Shortage**: Actual balance is lower than expected.
    
    Possible causes:
    - Cashier error (gave too much change)
    - Missing cash
    - Unrecorded void/refund
    - Counting error
  </Tab>
  
  <Tab title="Zero Difference">
    **Balanced**: Perfect match between expected and actual.
    
    This is the ideal scenario and indicates accurate transaction processing.
  </Tab>
</Tabs>

### Acceptable Tolerances

<Info>
  Most businesses set an acceptable variance threshold (e.g., ±$5 or ±0.5%) to account for minor counting errors or unavoidable discrepancies.
</Info>

## Best Practices

<Steps>
  <Step title="Count Opening Balance">
    Always count the opening balance with a witness and document it. Never assume the previous closing balance.
  </Step>
  
  <Step title="Reconcile Throughout the Day">
    For busy locations, consider mid-shift cash drops to reduce theft risk and closing time.
  </Step>
  
  <Step title="Document Variances">
    Create a log for significant variances with explanations and corrective actions.
  </Step>
  
  <Step title="Separate Cash and Cards">
    Always count and record cash and card totals separately for accurate reconciliation.
  </Step>
  
  <Step title="Secure Storage">
    Use a safe for opening balances and cash drops. Never leave large amounts in the register.
  </Step>
</Steps>

## Extending the Component

### Backend Integration

To connect to a backend service:

```typescript
// Add to constructor
constructor(private cashRegisterService: CashRegisterService) {}

// Load registers from API
ngOnInit(): void {
    this.loadCashRegisters();
}

loadCashRegisters() {
    this.cashRegisterService.getCashRegisters(this.companiaId).subscribe({
        next: (res) => {
            this.cashRegisters = res.data || [];
            this.updateActiveCashRegister();
        }
    });
}

// Save register opening
handleOpenCashRegister(): void {
    const payload = {
        companiaId: this.companiaId,
        userId: this.userId,
        openingBalance: this.openingBalance
    };
    
    this.cashRegisterService.openRegister(payload).subscribe({
        next: (res) => {
            this.cashRegisters = [res.data, ...this.cashRegisters];
            this.updateActiveCashRegister();
        }
    });
}
```

### Sales Integration

Update expected balance when sales occur:

```typescript
onSaleCompleted(invoice: Invoice) {
    if (this.activeCashRegister) {
        // Update expected balance based on payment methods
        const cashPayment = invoice.paymentMethods
            .filter(pm => pm.type === 'cash')
            .reduce((sum, pm) => sum + pm.amount, 0);
        
        this.activeCashRegister.expectedBalance += cashPayment;
        
        // Update in database
        this.cashRegisterService.updateExpectedBalance(
            this.activeCashRegister.id,
            this.activeCashRegister.expectedBalance
        ).subscribe();
    }
}
```

## Troubleshooting

<AccordionGroup>
  <Accordion title="Cannot open new register">
    Ensure the current register is closed first. Only one register can be open at a time in the current implementation.
  </Accordion>
  
  <Accordion title="Large variance detected">
    1. Recount the cash drawer
    2. Review the day's transactions for voids or refunds
    3. Check for unrecorded sales
    4. Verify card terminal batch totals match system records
  </Accordion>
  
  <Accordion title="Opening balance not saved">
    Check that the opening balance is greater than zero. The validation prevents opening with zero or negative amounts.
