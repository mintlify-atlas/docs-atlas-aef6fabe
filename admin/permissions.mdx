## Overview

The Permissions module provides comprehensive role-based access control (RBAC) for your POS system. Administrators can create roles, assign granular permissions across modules, and manage users with specific role assignments.

<Note>
  This module requires the **Permisos** permission with `canView` enabled. Editing roles and users requires `canCreate` and `canEdit` permissions.
</Note>

## Role Management Workflow

<Steps>
  <Step title="Access Permissions Module">
    Navigate to **Master > Permissions** from the main menu. The page displays two tabs: **Roles** and **Users**.
  </Step>

  <Step title="Create a New Role">
    Click the **New Role** button. A dialog opens with:
    - **Name**: Enter a descriptive role name (required)
    - **Description**: Explain the role's purpose (required)
    - **Active**: Toggle to activate/deactivate the role
    
    <Warning>
      Role names should be unique and descriptive (e.g., "Store Manager" instead of "Role1").
    </Warning>
  </Step>

  <Step title="Configure Module Permissions">
    For each module in the system, set granular permissions:
    
    | Permission | Description | Auto-Behavior |
    |-----------|-------------|---------------|
    | **View** | See module data | When disabled, all other permissions are revoked |
    | **Create** | Add new records | Requires View enabled |
    | **Edit** | Modify existing records | Requires View enabled |
    | **Delete** | Remove records | Requires View enabled |
    | **Export** | Export data to files | Requires View enabled |
    
    <Info>
      The system automatically disables Create, Edit, Delete, and Export when View is unchecked.
    </Info>
  </Step>

  <Step title="Save the Role">
    Click **Save**. The system:
    1. Validates that Name and Description are filled
    2. Creates the role record
    3. Saves all module permissions
    4. Displays success confirmation
    
    The role now appears in the roles list.
  </Step>

  <Step title="Edit Existing Role">
    Click the **Edit** icon on any role to modify:
    - Role name and description
    - Active status
    - Module permissions
    
    <Note>
      Changes to role permissions affect all users assigned to that role immediately.
    </Note>
  </Step>

  <Step title="Delete a Role">
    Click the **Delete** icon and confirm. 
    
    <Warning>
      Users assigned to deleted roles will lose their permissions. Reassign users before deleting.
    </Warning>
  </Step>
</Steps>

## User Management Workflow

<Steps>
  <Step title="Switch to Users Tab">
    Click the **Users** tab to view all system users. The table displays:
    - Full name
    - Email
    - Assigned role
    - Active status
    - Session status (currently logged in)
  </Step>

  <Step title="Create a New User">
    Click **New User** and fill in the form:
    - **First Name**: Required
    - **Middle Name**: Optional
    - **Last Name**: Required
    - **Second Last Name**: Optional
    - **Email**: Required (must be unique)
    - **Password**: Required (minimum 8 characters recommended)
    - **Role**: Select from active roles dropdown
    - **Status**: Active/Inactive toggle
  </Step>

  <Step title="Save User">
    Click **Save**. The system creates the user account with the assigned role's permissions.
    
    <Info>
      Users can log in immediately after creation using their email and password.
    </Info>
  </Step>

  <Step title="Edit User Details">
    Click **Edit** on any user to update:
    - Personal information
    - Email address
    - Assigned role
    - Active status
    
    <Note>
      Changing a user's role immediately updates their permissions across all modules.
    </Note>
  </Step>

  <Step title="Deactivate/Delete User">
    - **Deactivate**: Set Status to Inactive (preserves data)
    - **Delete**: Removes user from the system
    
    <Warning>
      Deleted users cannot be recovered. Consider deactivating instead.
    </Warning>
  </Step>
</Steps>

## Permission Cascading Logic

The system implements smart permission cascading:

```typescript
updateRolePermission(moduleId: number, key: keyof Permission, value: boolean) {
  this.currentRole.permissions = this.currentRole.permissions.map((p) => {
    if (p.moduleId === moduleId) {
      const updated = { ...p, [key]: value };
      
      // When View is disabled, revoke all other permissions
      if (key === 'canView' && !value) {
        updated.canCreate = false;
        updated.canEdit = false;
        updated.canDelete = false;
        updated.canExport = false;
      }
      return updated;
    }
    return p;
  });
}
```

<Info>
  This ensures users cannot have Create, Edit, Delete, or Export permissions without View access.
</Info>

## Permission Check Example

Components check permissions before rendering UI elements:

```typescript
ngOnInit(): void {
  const session = this.authService.getSession() as AuthSession;
  const { userId, companiaId } = session;
  
  this.loginService.getPermissions(userId, companiaId).subscribe({
    next: (permissions) => {
      this.permissions = permissions.data ?? [];
      const permission = this.permissions.find((p) => 
        p.module === 'Permisos'
      );
      
      this.canView = permission.canView;
      this.canCreate = permission.canCreate;
      this.canEdit = permission.canEdit;
      this.canDelete = permission.canDelete;
    }
  });
}
```

## Security Best Practices

<Warning>
  **Never grant excessive permissions.** Follow the principle of least privilege—users should only have permissions necessary for their job function.
</Warning>

<Info>
  **Regular audits:** Review user permissions quarterly to ensure they align with current responsibilities.
</Info>

<Note>
  **Test new roles:** Create a test user with the new role to verify permissions work as expected before assigning to production users.
</Note>

## Troubleshooting

### Users Can't Access Modules
- Verify the user's role has `canView` enabled for that module
- Check that the user's account status is Active
- Ensure the role itself is Active

### Permission Changes Not Applied
- Log out and log back in to refresh session permissions
- Check that permissions were saved successfully (watch for error toasts)
- Verify the correct role is assigned to the user

### Can't Delete Role
- Check if users are still assigned to the role
- Reassign users to a different role first
- Ensure you have `canDelete` permission for the Permisos module
